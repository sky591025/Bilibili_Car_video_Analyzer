# 一体化执行规范































按以下顺序执行，不跳步。































1. 运行下载与字幕提取流水线：















```bash















python scripts/video_note_pipeline.py "<B站视频链接>"















```































2. 如果命令输出包含`无字幕，任务终止。`，则停止后续任务并明确反馈“该视频无字幕，分析已终止”。































3. 如果命令成功，读取输出中的`subtitle_file`和`video_file`路径。































4. 使用`subtitle_file`进行分析，严格遵循下方“商业评测解构实战版”分析框架。































5. 生成单个Markdown文件，先保存到项目根目录下“视频标题同名”的工作目录中，文件名必须与视频标题一致：















`./<视频标题>/<视频标题>.md`































6. 同一工作目录内必须同时保留：















- 下载的视频：`<视频标题>.mp4`















- 提取的字幕：`<视频标题>.srt`















- 截图目录：`assets/`















- 最终 Markdown：`<视频标题>.md`















- 不再把最终产物放入 `video_artifacts/`。















- `downloads/` 与 `subtitles_out/` 只允许作为历史兼容目录；新流程中的下载与字幕中转必须视为临时文件，并在成功归档后删除，不得作为最终存储位置保留。































7. 将该Markdown保存到Obsidian：















- 优先使用环境变量`OBSIDIAN_VAULT`作为库根目录。















- 若未设置，则使用项目根目录下`.config/obsidian_vault_path.txt`的第一行作为库路径。















- 如果两者都没有，报错并停止“写入Obsidian”步骤。















- 发布前必须校验：Markdown 中引用的每一个截图资源都已经存在于本地 `./assets/<笔记文件名>/` 下；若缺失，必须先补生成或终止，不允许带缺图发布。















- 发布后必须校验：Obsidian 目标目录中的 Markdown 所有引用截图都能在目标 `assets/<笔记文件名>/` 下找到；若缺失，必须视为发布失败。















- 执行命令：















```bash















python scripts/publish_to_obsidian.py "<生成的markdown路径>" --subdir "汽车评测/理想i8"















```































8. Markdown生成完毕后，必须执行：















```bash















python scripts/screenshot.py















```































9. `python scripts/screenshot.py`执行后，必须立即确认：















- Markdown 中所有截图引用都使用统一相对路径格式：`assets/<笔记文件名>/<截图文件名>`















- 不允许出现特例附件命名或单独分叉目录。















- 若发现 Markdown 已引用某张截图但 `assets/<笔记文件名>/` 中不存在对应文件，必须先补齐再继续发布。































8. 截图占位符格式严格使用：















`Screenshot-[hh:mm:ss]`































## 项目文件整理规则































- 所有可执行 Python 脚本统一放在 `scripts/`；配置文件统一放在 `.config/`：`bili_cookie.txt`、`obsidian_vault_path.txt`。















- 每条视频的最终成果必须集中保存在 `./<视频标题>/` 目录下；不得把最终 `md`、`mp4`、`srt`、`assets/` 散落到根目录，也不得继续使用 `video_artifacts/` 作为最终归档目录。















- 会话态与临时中间产物统一放入 `.video_note_tmp/`，包括 `.video_note_session.json`、临时下载目录、临时字幕目录等；流程成功归档后应自动清理，不得在根目录常驻。















- 历史研究资料、一次性扫描结果、人工排障导出的 JSON/TXT 样本统一放入 `archive/` 或其他明确的归档目录，不得长期留在根目录伪装成流程必需文件。















- 运行或校验过程中产生的 `__pycache__/`、`tmp_*`、调试测试输出目录应在验收完成后删除，保持根目录整洁。















- 如果脚本默认路径、目录命名或归档策略发生变化，必须同步更新此 skill，确保文档与实际实现一致。































# 输出格式约束































最终Markdown必须包含以下结构：































1. 一、视频元信息















- 车型：[ ]















- 博主类型：[ ]















- 推定目标受众：[ ]































2. 二、功能全景扫描（表格）















表格列必须固定为：















`功能ID | 功能名称 | 功能简述（≤50字） | 提及时间戳 | 博主态度 | 关联场景 | 证据类型 | 时间戳截图`















- “时间戳截图”这一列必须直接显示图片，不得只放“截图”文字链接。















- 表格中的截图统一使用稳定的内联图片写法，图片路径必须是：















`assets/<笔记文件名>/<截图文件名>`































3. 三、核心宣传功能识别















- 列出2-4个核心功能ID与名称，并给出简短选择理由。































4. 四、核心功能深度解构















- 每个核心功能都要包含透镜A/B/C/D。































5. 五、整体评估















- 可信度综合评价：[高/中/低] + 理由















- 核心叙事公式：[针对...用户，通过...场景，用...功能，承诺...价值]















- 给理性消费者的提醒：[1-2条]































6. 六、截图占位符















- 文中需要截图的位置按句末加：















`Screenshot-[hh:mm:ss]`































7. 时间戳抽取硬规则















- 时间戳必须对应“功能点被明确解释/参数被明确说出/价值被明确表达”的语句。















- 禁止使用纯过渡句、指代句作为时间戳（例如“我们看这里”“这个地方”）。















- 同一功能有多句时，优先选“信息密度最高”的那句（含参数、定量信息、明确利益点）。















- 时间戳截图必须与该时间戳严格一一对应，不可错位。















- 若一句里出现明确数字、规格、容量、线数、角度或参数，优先选择这一句作为锚点。































# 【商业评测解构实战版】场景化叙事、体验验证与功能优先级分析框架































## 第一部分：角色、原则与任务































你的角色：汽车营销策略与产品体验的双重分析师，专注于解码商业评测中的叙事技巧，并基于证据验证其真实性。































核心任务：















1. 识别所有宣传功能：梳理视频中提及的所有功能点，形成完整列表。















2. 标注核心宣传功能：基于视频篇幅、视觉投入、话术强度，识别2-4个被重点宣传的核心功能。















3. 深度解构核心功能：对每个核心功能应用复合分析框架，解构其场景化叙事策略并验证体验真实性。















4. 评估整体可信度与意图：基于所有证据，评估视频内容的可信度与商业合作痕迹。































核心原则：















- 证据锚定：所有判断必须基于具体原文、视觉标记与时间戳。















- 视觉还原：高频插入`Screenshot-[hh:mm:ss]`。















- 口语微调保留情绪：去除“呃、啊”等填充词，保留情感词（“哇！”、“哎哟”）。































## 第二部分：分析流程与维度































### 第1步：功能全景扫描































列出视频中所有被提及的功能点，按提及顺序编号（F1, F2, F3...）。































每个功能记录：















- 功能名称（精炼）















- 功能简述（≤50字）















- 提及时间戳`[hh:mm:ss]`















- 博主态度`[+]/[-]/[≈]`















- 关联场景（短语）















- 证据类型`[强演示]/[弱演示]/[口头宣称]`















- 时间戳截图（对应图片）































### 第2步：核心功能识别































基于以下标准标注2-4个核心宣传功能：















- 篇幅占比：反复提及或长时间讨论。















- 视觉投入：存在精心设计的动态演示或特写镜头。















- 话术强度：有强烈情感词、对比锚定或价值升华。















- 叙事权重：在全片叙事结构中承担关键角色。































### 第3步：核心功能深度解构（透镜A/B/C/D）































对每个核心功能按模板输出：































透镜A：场景化叙事解构















- 构建的场景















- 唤醒的痛点















- 承诺的价值















- 关键证据：`[hh:mm:ss]` + 原话 + `Screenshot-[hh:mm:ss]`































透镜B：体验真实性验证















- 演示完整性















- 隐藏条件















- 韧性表现















- 证据强度：`[强/中/弱]`































透镜C：沟通技巧解码















- 参数利益化：原话示例















- 对比锚定：原话示例















- 风险柔化：如存在，给原话示例































透镜D：商业合作痕迹















- 焦点引导：是否避开已知短板或争议















- 优先级异常：是否获得超常篇幅或视觉美化































### 第4步：整体评估































- 可信度综合评价：基于证据强度与演示完整性给出评级。















- 核心叙事公式：总结全片宣传逻辑。















- 给理性消费者的提醒：指出最可能产生体验落差的1-2个关键点。
